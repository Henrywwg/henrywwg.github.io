Downscaler

Hardware-Accelerated Video Downscaler

To run the MediaPipe gesture recognition model efficiently on the Zynq Processing System, the 1080p camera feed must be downscaled to a 224x224 RGB format. Rather than relying on the ARM CPU for image processing, we implemented a custom RTL downscaler in the FPGA fabric to handle this step prior to memory storage.

Hardware Flow Diagram

Here we can see the abstracted view of the hardware/PL design. Generally, A MIPI CSI camera (PCAM5) clocked at 200MHz feeds into a demosaic module (Vivado IP) converting the raw bayer data into standard RGB. Here, the data stream splits. One stream is grayscaled, while the other stream remains RGB. The RGB stream is then fed into the downscaler (detailed below) while the grayscale stream is sent to the mDP output to aid framing the camera feed for the user.

Architecture and Data Flow

  The downscaler sits directly in the camera's AXI-Stream pipeline allowing us to avoid costly buffering in main memory until the final frame is ready.
Input: Raw 1080p video streamed directly from the camera sensor at a ~75MHz pixel clock.
Output: A 224x224 RGB frame (~150 KB) written to a /dev/shm RAM disk via a Video Direct Memory Access (VDMA) block.
Logic: The module utilizes a parameterizable adder tree for the downscaling math, requiring zero DSP slices.


Processing and Latency

 The downscaler's fabric is clocked at 200MHz giving us a theoretical compute time per frame of around 75 µs based solely on logic. Because we have to wait for the camera to stream in the pixels we are effectively limiting our processing time to around 33 ms due to the effective 75 MHz pixel clock. Additionally, since we do a fixed crop of only around the top 4/5's of the frame (since we crop only until the bottom 92 pixels), the compute is fully hidden within the camera's streaming time. Once the last line is processed tlast is asserted, tvalid is deasserted and the frame is committed to memory.


Software vs. Hardware Comparison

  • Hardware Pipeline: The camera reads in data at approximately 75MHz, but the hardware only requires the first 4/5th's of the frame to produce a properly downscaled 224x224 frame. This all happens in approximately 22.11 ms.
    (1920px * 1080px * (4/5)) /75MHz = 22.11 ms.
    The processing time of the downscaler is around 76 us total, but since it happens sequentially in the stream, we effectively hide the entire latency in the slower 75MHz camera stream. This means that we only use around 22 ms, along with the overhead of a single DRAM write of a small 224x224 image to save the final frame.


  • Estimated Software Pipeline: A standard software implementation would require the full 1080p frame to be buffered first in main memory (taking around 33ms to do so), before CPU downscaling can be performed, after which the downscaled frame can be committed to memory. This requires a total of 2 RAM writes (one for the full frame, one for the downscaled frame) and additional CPU overhead for downscaling.


Based on this analytical comparison, we can see that the hardware pipeline is much faster than a standard software implementation since, while the HW based approach relies on a single 224x224 RAM write (plus negligible processing time), the software approach requires a full 1080p frame RAM write, non-negligible downscaling time and then a second RAM write for the final frame. The HW approach saves us extra RAM writes and CPU overhead.
